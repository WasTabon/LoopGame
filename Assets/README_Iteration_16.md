# Loop Puzzle — Итерация 16: Порталы

Последняя новая механика на фундаменте итерации 11, плюс 3 демо-уровня (38–40). Это была самая хитрая по валидации механика — поэтому особое внимание корректности.

## Механика

**Портал** — это пара клеток с одинаковым id. Поток петли, дойдя до одного портала, **телепортируется** и продолжается из парного. То есть две клетки-портала логически «склеены»: вход в один = выход из другого.

- Портал — фиксированная клетка (не элемент, не вращается). У него есть «горло» (mouth) — направление, в котором он соединяется с обычным элементом.
- Поток входит в портал A со стороны его горла → выходит из горла портала B → продолжается дальше.
- Это позволяет петле «перепрыгивать» через поле, соединяя удалённые участки.

Сильно меняет пространственное мышление: петля больше не обязана быть непрерывной на поле — она может уходить в портал и появляться в другом месте.

## Демо-уровни (38–40)

Все построены по проверенной схеме «два сегмента, соединённые порталами в одну петлю»:
- **38** — два вертикальных сегмента, соединённых сверху и снизу двумя парами порталов (3×4).
- **39** — то же, сегменты дальше друг от друга (4×4).
- **40** — более длинные сегменты (3×5).

Игрок поворачивает элементы сегментов так, чтобы петля прошла через порталы и замкнулась.

## Как обеспечена корректность (особое внимание)

Модель портала спроектирована и проверена **сначала на Python**, и только потом перенесена в Unity — из-за сложности.

- Портал-валидатор (`ValidatePortals` в Unity / `is_portal_win` в Python): проверяет, что у всех элементов нет открытых концов (с учётом того, что портал принимает соединение со стороны горла), что все горла порталов состыкованы, и что трассировка с **телепортом** через порталы даёт замкнутую петлю.
- Ключевая логика трассировки: дойдя до портала со стороны горла, поток переходит к парному порталу и выходит из его горла.
- Проверено на эталонном кейсе: два раздельных сегмента, соединённых двумя парами порталов, дают ровно 1 замкнутую петлю.
- Каждый демо-уровень: решение — петля через порталы; старт не решён (сегменты разорваны); реконструкция (поворот в решение) выигрывает.
- Логика идентична в Python и Unity (трассировка половинных рёбер с телепортом).
- Структурная проверка: порталы парные (каждый id ровно дважды), на порталах нет элементов, ровно 1 стартовый маркер.
- Анимация потока на победе тоже проходит через порталы (телепорт в `LoopPathBuilder`).
- Все 37 предыдущих уровней (мосты, ломающиеся клетки, стрелки, цвета) по-прежнему валидны.

## Установка

1. Нужны выполненные Setup итераций 1–15.
2. Скопируй `Assets/LoopPuzzle/` поверх итерации 15.
3. Запусти `Tools → Loop Puzzle → Setup Iteration 16`.

Setup переимпортирует все 40 уровней (с полями портала и режима порталов), провалидирует, пересоберёт базу. Появятся уровни 38–40.

## Как проверить

- `Tools → Loop Puzzle → Validate All Levels` — все 40 уровней валидны.
- `Unlock All Levels`, открой **38/39/40** — увидишь круглые порталы (вращающиеся, парные по цвету). Поверни элементы так, чтобы поток ушёл в портал и вышел из парного, замкнув петлю. На победе анимация потока пройдёт через порталы.

## Заметки

- Портал нарисован кодом: кольцо с насечкой, указывающей направление горла; пара порталов окрашена одним цветом, медленно вращается.
- В демо порталы соединяют вертикальные сегменты — это чисто демонстрирует телепорт. Более хитрые конфигурации (петля с одним порталом-перехлёстом, порталы + другие механики) логично сделать в финальном ребалансе.
- Порталы — фиксированные клетки, заданные дизайном уровня; игрок их не двигает и не вращает.
